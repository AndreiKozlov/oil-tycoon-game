## Главный принцип: один бэкенд, много клиентов, ленивые данные

Игра — это **не Counter-Strike в реальном времени**. Это симулятор-стратегия. Это значит:

- Никаких 60 кадров/секунду от сервера
- Действия игрока — **редкие** (раз в 10-60 секунд: тапнул, построил, продал)
- Нет коллизий, нет физики, нет рендера на сервере
- Большинство «процессов» (добыча нефти, износ оборудования) — это **формулы**, а не симуляция

Это ставит нас в **очень комфортную нишу нагрузки** — как Clash of Clans или Forge of Empires, а не как PUBG.

## Почему сервер выдержит — конкретно

### 1. Lazy generation (ленивая генерация мира)

**Проблема:** карта эпохи на 30 000 участков. Если хранить каждый в БД с детальными данными — много места, медленные запросы.

**Решение:** участок существует в БД **только когда им владеют**.

Когда игрок смотрит на пустой участок, его параметры (тип нефти, объём, ландшафт) **рассчитываются на лету** из координат:

```javascript
function getEmptyPlotInfo(epochId, x, y) {
  // Детерминированный seed: одинаковые координаты → одинаковые свойства
  const seed = hash(`${epochId}-${x}-${y}`);
  return {
    landscape: pickLandscape(seed),
    oilEstimate: rng(seed, 0, 500_000),
    rarity: rollRarity(seed),
    // ...
  };
}
```

→ В БД лежат **только купленные участки** (тысячи, а не миллионы).

**Эффект:** база данных в **100-1000 раз** меньше, чем при честном хранении всех участков.

### 2. Tick-less экономика (нет таймеров)

**Плохой подход:** каждую секунду пробегать по всем игрокам и обновлять «добыто +0.1 баррель».
- 100 000 игроков × 10 вышек × 60 раз/сек = 60M операций/сек → **сервер падает**

**Хороший подход:** ничего не считается **до того момента**, когда игрок зашёл или продал.

Каждый объект в БД хранит:
- `lastUpdate` (timestamp)
- `productionRate` (баррелей/сек)
- `accumulatedBarrels` (буфер)

Когда игрок открывает игру или жмёт «забрать» — сервер делает **один расчёт**:
```
накоплено = (now - lastUpdate) × productionRate
```

→ Это **O(1)** на действие, а не **O(N)** в секунду.

Так работают **все** успешные мобильные симуляторы — от Township до Forge of Empires.

### 3. Кэш в Redis для горячих данных

В PostgreSQL не лазим по 100 раз за запрос. Кэшируем:
- **Профиль игрока** (TTL 5 мин или инвалидация при апдейте)
- **Лидерборды** (TTL 30 сек) — через Sorted Sets, **встроенный механизм Redis**
- **Цена нефти/газа сейчас** (мутируется по событию, читается всеми)
- **Карта эпохи** (генерируется раз, лежит вечно до конца эпохи)

Redis тянет **миллион запросов/сек** на одной машине. У нас столько и за день не будет.

### 4. Anti-cheat: вся логика на сервере

Клиент **никогда** не присылает «я добыл 1000 баррелей». Клиент присылает:
- «Я хочу построить вышку №42 на координатах X,Y»
- «Я хочу продать всё с участка №3»

Сервер сам:
- Проверяет что игрок может это сделать (деньги, лимит, технологии)
- Применяет действие
- Возвращает новое состояние

Это не только защита от читеров. Это и **производительность** — клиент не пересылает килобайты «состояния», только action + response.

### 5. Шардинг по эпохам (когда вырастем)

Каждая эпоха — это **отдельная карта** в БД. Можно хранить их **на разных серверах**.

- Эпоха 1 (старые игроки) — сервер А
- Эпоха 2 (новые игроки) — сервер Б
- Эпоха 3 (текущая активная) — сервер В

Это даёт **горизонтальное масштабирование** почти бесплатно — игроки одной эпохи не пересекаются с другой.

### 6. WebSocket — не для всех действий

WebSocket нужен **только** для:
- Цены нефти в реальном времени (push 1 раз в 5-10 сек)
- Новые позиции в лидерборде (push при изменении топ-100)
- Мировые события (push когда стартует ивент)
- Уведомления (взломан склад, разлив)

**Покупка/строительство/продажа — REST**. Они нечастые, их не нужно держать в открытом коннекте.

Один WebSocket-сервер легко держит **10 000 одновременных коннектов** на средней машине.

## Где могут возникнуть проблемы

Чтобы не быть розовым оптимистом, перечислю **реальные узкие места**.

### 1. Дудосы и боты (90% всех «нагрузок»)

P2E игры **гарантированно** привлекают:
- Скрипты которые регают тысячи аккаунтов
- Боты которые тапают вместо людей
- DDoS от завистников/конкурентов

**Защита (с первого дня):**
- Cloudflare bot protection (бесплатно)
- Rate limiting на API (10-50 запросов в минуту на IP)
- Привязка к Telegram ID (один Telegram = один аккаунт)
- Подозрительная активность → шадоу-бан (игрок не знает что забанен, продолжает играть, но в изолированной песочнице)

### 2. Резкий вирусный рост (хорошая проблема)

Если попадаешь в тренды и за ночь приходит 1M человек:
- Сервер падает, очередь регистрации
- Тысячи злых твитов
- Конкуренты в радости

**Защита:**
- Cloudflare держит входящий трафик и показывает страницу «вы в очереди, осталось N минут»
- Подъём дополнительных серверов из готовых snapshot'ов (15-30 мин)
- Очередь регистрации на старте каждой эпохи

### 3. Состояние гонки в экономике (race conditions)

Игрок одновременно из 2 устройств жмёт «купить участок». Сервер списывает деньги **дважды** или продаёт участок **двум**.

**Защита:**
- Транзакции БД (SELECT FOR UPDATE)
- Оптимистическая блокировка на уровне игрока (version field)
- Идемпотентные запросы (с уникальным client_request_id)

### 4. Чит «изменю время на телефоне»

Игрок переводит часы на 100 дней вперёд → «добыча накопилась» → продаёт миллион баррелей.

**Защита:**
- Время **только серверное**. Клиент не сообщает время.
- Все таймеры рассчитываются от серверного `lastUpdate` к серверному `now`.
- Сверка серверного и клиентского времени на каждый запрос → если расхождение > 5 мин, действие отклоняется.

### 5. Утечки памяти и GC-паузы в Node.js

Если бэкенд на Node.js — он может «замерзать» на сборке мусора при больших аллокациях.

**Защита:**
- Профилирование с первого дня (Clinic.js, 0x)
- Перезагрузка процессов каждые N часов (PM2 grace restart)
- Если совсем больно — переезд критичных эндпойнтов на Go или Rust

## Алгоритм действий когда сервер ляжет

Это **обязательно произойдёт**. Готовимся.

1. **Health-чек** мониторинга → автоматический алерт в Telegram через 30 сек
2. **Cloudflare** включает «under attack mode» — показывает страницу-капчу
3. **Логи** в Sentry показывают где упало
4. **Откат деплоя** (если связан с релизом) — `git revert` + redeploy за 5 минут
5. **Подъём резервного сервера** из snapshot'а — 15-20 минут
6. **Объявление в Telegram-канале** — «работаем, скоро вернёмся»

**Готовый runbook** (документ что делать при падении) — пишется **до** первого падения, не после.

## Финальная мысль

Серверная нагрузка — это **самое предсказуемое** что есть в проекте. Гораздо страшнее то, что нельзя предугадать: понравится ли игра людям, окупится ли реклама, не уйдут ли все через неделю.

Если игра **понравится** — деньги на сервера будут с первого месяца.
Если **не понравится** — серверов хватит и одного дешёвого.

**Правило:** не оптимизируем до того, как есть проблема. На MVP — один VPS, простой код, понятная архитектура. Растём — масштабируем по факту, не на бумаге.

## Интеграция с биржей (реальные цены нефти и газа)

### Цель
Игровые цены подвязаны к реальным мировым ценам Brent, WTI, Natural Gas. Каждый час подтягиваем свежие данные, кэшируем, игра работает на кэше.

### Источники
| Источник | Что берём | Лимит | Цена |
|----------|-----------|-------|------|
| **Yahoo Finance** | BZ=F (Brent), CL=F (WTI), NG=F (Natural Gas) | 2000 req/час | $0 |
| **EIA API** | бэкап для США | с ключом, 5000/день | $0 |
| **MOEX** | Urals, российский газ | публичные данные | $0 |

Используем библиотеку `yfinance` (Python) или `yahoo-finance2` (Node.js). Никакого скрейпинга HTML — официальный API.

### Архитектура потока данных

```
┌─────────────────────────────────────────────────┐
│ CRON (раз в час)                                │
│                                                 │
│  → fetch("BZ=F", "CL=F", "NG=F")                │
│  → парсим: { brent, wti, gas }                  │
│  → считаем производные:                         │
│       urals  = brent × 0.85                     │
│       dubai  = brent × 0.97                     │
│       heavy  = brent × 0.75                     │
│  → Redis SET prices:current { ...все... }       │
│  → Redis ZADD prices:history (timestamp)        │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│ GAME API (любой запрос на продажу)              │
│                                                 │
│  1. GET prices:current из Redis (1 ms)          │
│  2. Применить игровые модификаторы:             │
│       событие сезона (Redis SET events:current) │
│       микро-колебание (синусоида от времени)    │
│       региональный множитель                    │
│  3. Финальная цена → продажа                    │
│  4. Записать транзакцию в PostgreSQL            │
└─────────────────────────────────────────────────┘
```

### Принципы устойчивости

**Игра НЕ зависит от живого API биржи.**

1. **Cron** идёт за ценами раз в час, в фоне.
2. **Игра** всегда читает из Redis. Yahoo может лежать сутки — игра работает.
3. **TTL кэша 25 часов** — даже если API упало на день, есть последние данные.
4. **Если кэш старше 48 часов** — алерт админу + игра переходит на дефолтные цены (Brent = $70).
5. **При первом запуске** — пока cron не отработал, играем на дефолтах.

### Структура хранения

**Redis:**
```
prices:current          → JSON { brent: 72.4, wti: 69.8, gas: 2.34, urals: 61.5, ... }
prices:current:updated  → timestamp последнего обновления
prices:history:brent    → Sorted Set (timestamp → price), хранит последние 30 дней
prices:history:gas      → то же для газа
events:current          → активное мировое событие { type, multiplier, ends_at }
```

**PostgreSQL** (только для аналитики/истории):
```sql
CREATE TABLE price_snapshots (
  id BIGSERIAL PRIMARY KEY,
  ts TIMESTAMPTZ NOT NULL,
  brent NUMERIC(10,2),
  wti NUMERIC(10,2),
  gas NUMERIC(10,4),
  urals NUMERIC(10,2),
  dubai NUMERIC(10,2),
  heavy NUMERIC(10,2)
);
CREATE INDEX idx_price_ts ON price_snapshots(ts DESC);
```

→ Хранит историю на годы для графиков в игре.

### Игровая формула цены при продаже

```javascript
function calculateSellPrice(oilType, baseQuantity) {
  // 1. Базовая цена из Redis (реальная биржа)
  const realPrice = redis.get(`prices:current.${oilType}`);

  // 2. Активное мировое событие
  const event = redis.get('events:current');
  const eventMult = event ? event.multiplier : 1.0; // например 1.25 за «авария»

  // 3. Микро-колебание (синусоида от часа дня + случайность)
  const microFluct = 1.0 + 0.2 * Math.sin(Date.now() / 1800000) + (Math.random() - 0.5) * 0.1;

  // 4. Финальная цена
  const finalPrice = realPrice * eventMult * microFluct;

  return Math.round(finalPrice * baseQuantity);
}
```

### Дополнительные фичи на этой инфраструктуре

**Графики цен в игре.** Из `prices:history` рисуем линейный график за неделю. Реальные данные, обновляются каждый час. Это образовательный элемент + залипательная механика.

**Новостной фид.** Раз в день парсим заголовки из EIA или OilPrice.com → показываем в игре как «мировые новости». Игрок понимает почему в игре цена скачет.

**Сезонность газа автоматически.** Зимой в реальном мире газ дороже → в игре тоже. Без специальных кодов — просто потому что цены реальные.

### Правовая сторона

✅ **Использование реальных биржевых цен — легально.** Это публичная информация, не защищена авторским правом. Yahoo Finance free tier разрешает некоммерческое и низкое коммерческое использование.

❌ **Нельзя:** конвертировать игровую нефть в реальные деньги по реальной цене (это будет нефтяной фьючерс/дериватив, регулируется).

✅ **Можно:** игровая нефть продаётся за игровую валюту, цена которой привязана к реальной. Это просто «контент с открытыми данными» — как любая игра использующая Bloomberg-стиль котировок.

### Стоимость и сложность

- **Инфра:** $0 (Yahoo бесплатно, Redis уже есть в стеке)
- **Код:** ~200 строк (cron + парсинг + Redis + формула)
- **Время на разработку:** 1-2 дня
- **Поддержка:** минимальная — мониторим что cron живой и кэш свежий

## Парсинг реальных новостей и Event Engine

### Цель
Игровые события **частично триггерятся реальными новостями** из мира нефтегаза. Это уникальная фишка проекта.

### Источники RSS-фидов

| Источник | URL | Что даёт |
|----------|-----|----------|
| **Reuters Energy** | reuters.com/business/energy/rss | топ-новости индустрии |
| **OilPrice.com** | oilprice.com/rss | специализированные новости |
| **Bloomberg Markets** | bloomberg.com/markets/energy/rss | макро-аналитика |
| **EIA News** | eia.gov/rss | официальные данные США |
| **OPEC News** | opec.org/opec_web/en/feed | официальные пресс-релизы OPEC |
| **TASS Energy** | tass.ru/rss/energetika.xml | российские новости (RU локаль) |

Все источники — **публичные RSS, бесплатные**.

### Поток данных

```
┌─────────────────────────────────────────────────┐
│ CRON каждый час                                 │
│                                                 │
│  for each RSS feed:                             │
│    fetch(feed_url)                              │
│    parse XML → list of articles                 │
│    for each article:                            │
│      if article.id NOT in db.processed_news:    │
│        classify(article)                        │
│        if matches_trigger:                      │
│          create_event(trigger_type)             │
│        save_to_db(article)                      │
└─────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│ EVENT ENGINE                                    │
│                                                 │
│  • Создаёт игровое событие (Redis events:*)     │
│  • Шлёт push всем активным игрокам              │
│  • Логирует в news_events таблицу для аудита    │
└─────────────────────────────────────────────────┘
```

### Классификация новостей

**Простой подход (старт):** регулярные выражения по ключевым словам.

```javascript
const TRIGGERS = {
  'opec_cut': /OPEC.{0,30}(cut|reduce|сократи)/i,
  'sanctions': /(sanction|санкци|embargo)/i,
  'cold_winter': /(cold|winter|freeze|мороз).{0,40}(europe|европ)/i,
  'oil_spill': /(oil spill|разлив нефти|leak|авария)/i,
  'hurricane': /(hurricane|storm|ураган|тайфун)/i,
  'discovery': /(discovery|найдено|новое месторождение|new field)/i,
  'crisis': /(oil crisis|нефтяной кризис|energy surge)/i,
};

function classify(article) {
  const text = `${article.title} ${article.description}`;
  for (const [trigger, regex] of Object.entries(TRIGGERS)) {
    if (regex.test(text)) return trigger;
  }
  return null;
}
```

**Продвинутый подход (если будет бюджет):** LLM-классификатор (Claude/GPT с дешёвой моделью) для точности и контекста. Стоимость ~$0.001 за статью, при 200 статей/день = $6/мес.

### Защита от перегрузки

```javascript
const MAX_TRIGGERS_PER_DAY = 3;
const TRIGGER_COOLDOWN = 6 * 60 * 60 * 1000; // 6 часов между триггерами одного типа

function shouldTrigger(triggerType) {
  const todayCount = redis.get('news:triggers:today:count') || 0;
  if (todayCount >= MAX_TRIGGERS_PER_DAY) return false;

  const lastFired = redis.get(`news:triggers:last:${triggerType}`);
  if (lastFired && Date.now() - lastFired < TRIGGER_COOLDOWN) return false;

  return true;
}
```

Если новостей нет или они отфильтрованы — `synthetic event scheduler` запускает события из встроенных таблиц GDD §21. Игра не зависит от внешних источников.

### Защита от провокаций (важно)

Чёрный список — **не** триггерим события связанные с:
- Конкретными военными конфликтами (имена стран в конфликте)
- Политиками поимённо
- Конкретными компаниями (избегаем юридических претензий)
- Религиозными темами
- Катастрофами с человеческими жертвами

При обнаружении таких слов в новости — пропускаем без триггера, **не** показываем игрокам.

**Killswitch:** в админке — кнопка «остановить парсинг новостей на 24 часа». Если в реальном мире что-то серьёзное — отключаем, ставим уважительное системное сообщение.

### Структура БД

```sql
CREATE TABLE processed_news (
  id VARCHAR(255) PRIMARY KEY, -- хэш URL или GUID статьи
  source VARCHAR(64) NOT NULL,
  title TEXT NOT NULL,
  url TEXT,
  published_at TIMESTAMPTZ,
  processed_at TIMESTAMPTZ DEFAULT now(),
  trigger_type VARCHAR(64), -- NULL если не сработал триггер
  raw_data JSONB
);
CREATE INDEX idx_news_processed ON processed_news(processed_at DESC);
CREATE INDEX idx_news_trigger ON processed_news(trigger_type) WHERE trigger_type IS NOT NULL;

CREATE TABLE game_events (
  id BIGSERIAL PRIMARY KEY,
  event_type VARCHAR(64) NOT NULL,
  level VARCHAR(16) NOT NULL, -- micro / local / global / tournament / epoch_end
  triggered_by VARCHAR(32),   -- 'news', 'scheduled', 'manual', 'synthetic'
  source_news_id VARCHAR(255) REFERENCES processed_news(id),
  payload JSONB NOT NULL,     -- эффекты, длительность
  starts_at TIMESTAMPTZ NOT NULL,
  ends_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_events_active ON game_events(starts_at, ends_at);
```

### Стоимость и сложность

- **Инфра:** $0 (RSS бесплатно, парсинг на том же сервере)
- **LLM-классификатор (опционально):** ~$5-10/мес при включении
- **Код:** ~400-600 строк (парсер + классификатор + event engine + защиты)
- **Время на разработку:** 3-5 дней
- **Поддержка:** мониторим что cron живой, проверяем логи триггеров раз в неделю, обновляем чёрный список по необходимости

### Что это даёт

- **Уникальное позиционирование** — «игра реагирующая на реальный мир»
- **Бесконечный поток контента** — мир сам генерирует события
- **Образовательный элемент** — игроки узнают о реальной индустрии
- **Маркетинговый ход** — Telegram-канал игры может постить «реальная новость → реакция в игре»
- **Эмерджентность** — невозможно «решить» игру, всегда есть свежие события
