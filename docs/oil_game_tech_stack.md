## Главные принципы выбора стека

1. **Один язык на фронте и бэке** — TypeScript. Экономит мозг, общие типы.
2. **Один код на все платформы** — Web / Telegram / iOS / Android из одной кодовой базы.
3. **Только проверенные технологии** — то что работает у тысяч проектов. Никаких экспериментов.
4. **Минимум зависимостей на старте** — каждая библиотека = риск. Берём только то что реально нужно.
5. **Возможность масштабирования** — не упрёмся в потолок при росте DAU.
6. **Дешёвая инфраструктура** — Hetzner вместо AWS.

## Структура проекта (монорепо)

```
oil-game/
├── apps/
│   ├── web/                 # React-фронтенд (Web + Telegram Mini App)
│   │   ├── src/
│   │   ├── package.json
│   │   └── vite.config.ts
│   └── api/                 # Fastify-бэкенд
│       ├── src/
│       ├── package.json
│       └── drizzle.config.ts
├── packages/
│   ├── shared/              # общие типы и константы
│   │   ├── src/
│   │   └── package.json
│   ├── game-engine/         # игровая логика (формулы, баланс)
│   │   ├── src/
│   │   └── package.json
│   └── ui/                  # переиспользуемые UI-компоненты
│       └── package.json
├── docs/                    # GDD, balance, architecture
├── pnpm-workspace.yaml      # настройка монорепо
├── tsconfig.base.json       # общая TS-конфигурация
└── package.json
```

**Зачем монорепо:** общий код (типы, формулы баланса) используется и на фронте, и на бэке. Меняется в одном месте → автоматически обновляется везде.

## Альтернативы которые отвергли (и почему)

### Языки бэка

| Что | Почему НЕ выбрали |
|-----|-------------------|
| Python (FastAPI) | разрыв с фронтом, нет общих типов |
| Go | быстрее, но дольше писать, для одиночки |
| Rust | топ-перформанс, но сложно и медленно разрабатывать |
| PHP | устаревший подход для real-time игры |
| Ruby on Rails | дорого хостить, маленькое комьюнити для игр |
| Java/Kotlin (Spring) | overengineered для нашего размера |

### Базы данных

| Что | Почему НЕ выбрали |
|-----|-------------------|
| MongoDB | NoSQL хуже для финансовых транзакций |
| MySQL | Postgres лучше по фичам (JSON, partial indexes) |
| SQLite | только для прототипа, не выдержит multiplayer |
| Firebase Firestore | дорого при росте, vendor lock |
| DynamoDB | AWS lock-in, сложный pricing |

### Игровые движки

| Что | Почему НЕ выбрали |
|-----|-------------------|
| Unity | тяжёлый для веб, не нужен для 2D |
| Godot | хорош, но Web-экспорт сырой |
| Cocos2d-js | менее популярен чем Pixi |
| Construct 3 | визуальное программирование — для непрофессионалов |
| Babylon.js | для 3D, у нас 2D |

### Хостинг

| Что | Почему НЕ выбрали |
|-----|-------------------|
| AWS | в 5-10 раз дороже Hetzner, сложный pricing |
| Google Cloud | то же, дороже |
| DigitalOcean | хорош, но Hetzner ещё дешевле |
| Railway / Render | удобно, но дорого при росте |
| Heroku | устарел, дорого |

## Карта обучения для разработчика

Если ты программист но не работал с этим стеком — порядок изучения:

1. **TypeScript** — основы типов, generics (1 неделя)
2. **React + hooks** — useState, useEffect, useMemo (1 неделя)
3. **Vite** — настройка проекта (1 день)
4. **Tailwind CSS** — utility-classes (3 дня)
5. **Zustand** — простой state management (1 день)
6. **TanStack Query** — кеш и запросы (2 дня)
7. **Pixi.js** — спрайты, контейнеры, тикер (1 неделя)
8. **Node.js + Fastify** — роуты, plugins (3 дня)
9. **PostgreSQL + Drizzle** — схема, миграции, запросы (1 неделя)
10. **Redis + ioredis** — кэш, sorted sets (3 дня)
11. **Socket.io** — события клиент-сервер (3 дня)
12. **Telegram Mini App SDK** — авторизация, Stars (3 дня)

**Итого:** ~6-8 недель плотного обучения с нуля до уверенного владения.

Если LLM помогает (как сейчас) — в 2 раза быстрее.

## Что дальше

Этот документ — техническая основа. Следующий шаг:

1. **Установить локальное окружение** (Node, pnpm, Postgres, Redis) — ~30 минут
2. **Создать пустой проект** (монорепо) — ~30 минут
3. **Поднять первый «Hello, World»** — фронт показывает данные с бэка — ~1 час
4. **Начать первый экран** — стартовая страница с выбором языка

Всё это — **этап 1, шаги 1-3 из roadmap**.

Когда будет нужно — детально пройдёмся по каждому шагу с командами и кодом.
